# Releasing Motif

How Motif versions, tags, GitHub Releases, and Chrome Web Store submissions fit together.

## Overview

```text
feature branch
  bump package.json (npm version … --no-git-tag-version)
  commit extension changes (pre-commit enforces bump)
  PR → merge to main
        ↓
GitHub Release workflow (if version > latest v* tag)
  npm run zip
  LLM release notes
  tag v{version} + GitHub Release + zip asset
        ↓
Manual Chrome Web Store upload
```

Direct pushes to `main` are blocked. All releases land via PR merge.

---

## Version source of truth

| What | Where |
|------|-------|
| **Canonical version** | `package.json` → `"version"` |
| **Built manifest** | WXT copies `package.json` into `.output/chrome-mv3/manifest.json` on `npm run build` / `npm run zip` |
| **Git tag** | `v{version}` (e.g. `1.0.3` → `v1.0.3`) |
| **GitHub Release** | `Motif v{version}` with attached `motif-{version}-chrome.zip` |
| **Chrome Web Store** | Same version string as `package.json` |

Do **not** set a separate version in `wxt.config.ts`. WXT treats `package.json` as the only source ([WXT manifest docs](https://wxt.dev/guide/essentials/config/manifest)).

## When to bump

Bump **`package.json`** before merging a PR that changes **shippable extension code**:

- `entrypoints/`
- `components/`
- `features/`
- `hooks/`
- `lib/`
- `utils/`
- `public/`
- `assets/`
- `wxt.config.ts`

You **do not** need a bump for docs-only, specs, CI workflow-only, or README changes on the same version.

### Semver guidance

| Change type | Example | Command |
|-------------|---------|---------|
| Bug fix | Vocab punctuation strip | `npm version patch --no-git-tag-version` |
| New user-facing feature | Live transcript export | `npm version minor --no-git-tag-version` |
| Breaking change | Removed permission, incompatible settings | `npm version major --no-git-tag-version` |

Always use **`--no-git-tag-version`**. That updates `package.json` and `package-lock.json` only. **Do not** create `v*` tags locally — CI creates the tag when the merge lands on `main`.

## Pre-commit guard

After `npm install`, **simple-git-hooks** registers a pre-commit hook that runs:

```bash
npm run check-version
```

Implemented in `scripts/check-version.mjs`. On every commit it:

1. `git fetch --tags origin` (best effort)
2. Reads `package.json` version
3. Finds the latest merged `v*` tag on the current branch history
4. **Passes** if there is no remote tag yet, or if `package.json` is **greater than** the latest tag
5. **Passes** if the version equals the latest tag but staged files are **not** release-relevant (docs-only)
6. **Fails** if extension code is staged without a version bump above the latest tag

Debug locally:

```bash
npm run check-version
```

## GitHub Release workflow

File: [`.github/workflows/release.yml`](.github/workflows/release.yml)

**Trigger:** push to `main` (including after PR merge).

**Steps:**

1. `npm ci`
2. `node scripts/release-version.mjs` — decides whether to release
3. If `should_release` is false → exit (docs/infra merge at same version)
4. If true → `npm run zip`
5. `node scripts/generate-release-notes.mjs` — LLM notes (see below)
6. Publish GitHub Release via `softprops/action-gh-release`

**Release condition** (`scripts/release-version.mjs`):

- `package.json` version is **greater than** the latest merged `v*` tag, **and**
- tag `v{version}` does **not** already exist

Dry run locally:

```bash
node scripts/release-version.mjs
```

Example output when current version matches latest tag:

```json
{
  "version": "1.0.2",
  "from_tag": "v1.0.2",
  "should_release": "false"
}
```

### Release notes

Generated by `scripts/generate-release-notes.mjs` using **OpenAI `gpt-4o-mini`**, based on commits and diff since the previous tag.

**Requires:** repository secret `OPENAI_API_KEY` (Settings → Secrets and variables → Actions).

**Optional override:** add `releases/{version}.md` before merge (e.g. `releases/1.0.3.md`). If present, CI uses that file instead of the LLM.

## Chrome Web Store

GitHub Releases produce the zip and public release notes. **Store upload is still manual:**

1. Open the [Chrome Web Store Developer Dashboard](https://chromewebstore.google.com/devconsole)
2. Download `.output/motif-{version}-chrome.zip` from the GitHub Release (or run `npm run zip` locally at the tagged commit)
3. Upload as a new version; paste release notes from GitHub into “What’s new” if useful

Published listing: [Motif on Chrome Web Store](https://chromewebstore.google.com/detail/motif/mlgmlehgbmoebpgghjnkheaciijcfblo)

After upload, Chrome pushes updates to installed users automatically (typically within hours of approval).

## Release checklist

### Before merge

- [ ] Bump version on the feature branch (`npm version patch|minor|major --no-git-tag-version`)
- [ ] Extension changes + version bump committed; pre-commit passes
- [ ] PR reviewed and merged to `main` via GitHub (not direct push)

### After merge

- [ ] **Actions → Release** workflow succeeded (or skipped intentionally for non-release PR)
- [ ] Tag `v{version}` exists on GitHub
- [ ] GitHub Release has release notes + `motif-{version}-chrome.zip` attached
- [ ] Smoke-test the zip (see below)
- [ ] Upload zip to Chrome Web Store
- [ ] Store review approved; verify version on listing

### Pre-submit smoke (recommended)

From [specs/testing.md](specs/testing.md):

- Fresh profile install
- First-run model download completes
- Option+S on selected French text → overlay + playback
- Option+T on a media tab → transcript overlay
- No console errors on a news article and YouTube
- Manifest permissions and privacy policy still accurate

Also verify: store listing assets, permissions justification, privacy policy URL.

## FAQ

**Why did CI skip the release after my merge?**

`package.json` version was not higher than the latest `v*` tag, or tag `v{version}` already exists. Infra/docs-only merges are expected to skip.

**Do I tag locally after merging?**

No. The Release workflow creates `v{version}` when the version bump merges to `main`.

**Can I release without OpenAI?**

Add `releases/{version}.md` before merge, or temporarily disable the generate-notes step. The workflow expects `OPENAI_API_KEY` when no override file exists.

**What about branch-only tags like `v1.1.0` on a feature branch?**

Only tags **merged into `main` history** count (`git tag -l "v*" --merged HEAD`). Tags on unmerged branches are ignored by the guard and release logic.

**How do users get the update?**

Chrome Web Store installs update automatically after Google approves the new package. Users can force-check at `chrome://extensions` → **Update**.
